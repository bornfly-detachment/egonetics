# Notion风格块编辑器 — VibeCoding避坑核心文档

> 目标：让AI帮你实现时不走弯路，只讲精髓和坑。

---

## 一、最核心的认知

**万物皆Block，Block可以嵌套Block。**

```
一个页面 = 一个Block
页面里的每一行 = 一个Block
一行里的加粗文字 = Block的属性，不是独立Block
一个折叠列表 = 一个Block，它的子内容也是Block
```

这个认知决定了所有的数据结构和交互逻辑。

---

## 二、数据结构精髓

### 只需要一张表，自引用

```
Block {
  id          唯一ID
  parent_id   父Block的ID，顶层为空
  type        block类型
  content     富文本内容（结构化JSON）
  position    排序位置（浮点数！）
}
```

### position必须用浮点数，不能用整数

```
原因：在两个Block之间插入时，取中间值，永远不需要更新其他Block

初始状态：A=1.0  B=2.0  C=3.0
A和B之间插入D：D=1.5
再插入E：E=1.25

如果用整数，每次插入都要重排所有Block的序号，性能灾难。
```

### content存富文本片段数组，不存HTML

```json
{
  "rich_text": [
    { "text": "普通文字", "bold": false, "italic": false },
    { "text": "加粗内容", "bold": true,  "italic": false },
    { "text": "红色文字", "bold": false, "color": "red" }
  ]
}
```

**为什么不存HTML：**
- HTML跨端渲染不一致
- 无法做精确的diff和撤销
- 富文本片段数组可以精确控制每个字的样式

## 三、前端状态管理精髓

### 用扁平Map存储，不用树形结构

```
错误做法：
{
  id: "root",
  children: [
    { id: "A", children: [...] }
  ]
}

正确做法：
Map {
  "root" → Block,
  "A"    → Block,
  "B"    → Block,
}
```

**为什么：**
- 树形结构更新一个节点需要深度遍历
- Map直接通过ID查找，O(1)
- 树形关系通过parent_id动态计算，不需要维护

### getChildren只做一件事

```javascript
getChildren(parentId) {
  return 所有Block中
    .filter(b => b.parent_id === parentId)
    .sort((a, b) => a.position - b.position)
}
```

渲染时调用这个函数，递归渲染子节点，就是完整的树。

## 四、编辑器内核的坑

### 绝对不要自己写contentEditable

contentEditable是浏览器原生的富文本编辑区域，问题极多：
- 不同浏览器行为不一致（回车产生div还是p，各浏览器不同）
- 中文输入法（IME）有大量边界case
- 光标位置计算极其复杂
- 粘贴内容带外部样式污染

**让AI去接入成熟的开源编辑器内核，而不是自己实现。**

## 五、键盘交互规则（必须严格实现）

这是产品体验的核心，每一条都不能随意改动：

```
Enter键
  → 在当前Block后面，插入同级新Block
  → 如果当前是标题或待办，新Block类型改为普通文本
  → 不是在当前Block内换行（这是最反直觉的地方）

Backspace键（当内容为空时）
  → 删除当前Block
  → 光标移到上一个Block的末尾
  → 内容不为空时，正常删除字符

Tab键
  → 当前Block变成上一个Block的子Block
  → parent_id改为上一个Block的id
  → position设为上一个Block子节点的末尾

Shift+Tab键
  → 当前Block从父Block里移出
  → 变成父Block的下一个同级Block
  → parent_id改为祖父Block的id

/键（斜杠命令）
  → 弹出类型选择菜单
  → 用户选择后，改变当前Block的type
  → 清除输入的斜杠字符
```

## 六、网络同步策略

### 本地状态和远端状态必须分离

```
用户按键
  ↓
立即更新本地Map（用户零延迟感知）
  ↓
把变更放入待发送队列
  ↓
500ms内没有新操作，批量发送给后端
  ↓
后端返回成功，什么都不用做
后端返回失败，回滚本地状态，提示用户
```

**绝对不能：** 等后端返回才更新界面，用户打字会感觉卡顿。

### 操作要批量合并再发送

```
用户连续打了"hello"5个字
产生5次content更新
合并成1次请求发送
不合并就是5倍请求量
```

## 七、后端接口设计原则

### 接口设计要基于"操作"，不是"状态替换"

```
错误设计：
PUT /blocks/:id  把整个block传过去替换

正确设计：
POST /blocks/operations  提交发生了什么操作
```

**为什么：**
- 操作可以记录历史，支持撤销
- 操作可以广播给其他设备，支持多端同步
- 状态替换在并发时会互相覆盖

### 删除Block必须级联删除所有子Block

```
删除一个toggle Block
必须同时删除它下面所有层级的子Block
否则数据库里留下大量孤儿数据
查询时也会出现幽灵Block
```

---

## 八、性能关键点

### 懒加载，不要一次加载整个文档

```
打开页面：只加载顶层Block
折叠的toggle展开时：才加载子Block
数据库类型Block：只加载当前屏可见的行
```

文档可能有几千个Block，一次全加载是性能灾难。

### 虚拟滚动（文档很长时才需要）

```
只渲染当前屏幕可见区域的Block
上下各留一屏的缓冲
滚动时动态换入换出
其他Block只占位，不创建DOM
```

这是后期优化项，先做功能，遇到性能问题再加。

## 九、最容易踩的坑汇总

```
❌ position用整数
✓  position用浮点数，插入取中间值

❌ 前端用树形结构存Block
✓  用扁平Map，通过parent_id动态建树

❌ content存HTML字符串
✓  存富文本片段数组

❌ 用户打字就发请求
✓  本地立即更新，500ms防抖后批量发送

❌ 自己实现contentEditable编辑器
✓  接入成熟的编辑器内核库

❌ 删除Block只删一条数据库记录
✓  级联删除所有子Block

❌ 上来就做多人实时协作
✓  先做好单用户完整体验，协作是另一个量级的复杂度

❌ 一次性加载整个文档所有Block
✓  懒加载，展开时才加载子节点
```

## 十、实现顺序建议

```
第一步：把Block的增删改查和嵌套关系跑通
        验证position浮点数插入逻辑正确

第二步：把键盘交互的Enter/Backspace/Tab实现对
        这是体验的核心，要仔细测试边界情况

第三步：加防抖同步，把本地状态和远端分离

第四步：加斜杠命令，支持切换Block类型

第五步：加拖拽排序

第六步：性能有问题了再加懒加载和虚拟滚动
```

不要跳步，每一步都要稳定后再做下一步。
